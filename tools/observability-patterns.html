You Can't Fix What You Can't See

A complete guide to the 6 observability patterns every microservices engineer must know

📖 8 min read · 🎯 Mid → Senior Engineer · 🏗️ Production Systems

In a monolith, debugging is uncomfortable. In microservices, without proper observability, debugging is nearly impossible. Your logs are in 12 places. The failing request touched 6 services. And your on-call alert fired at 2am.

This post breaks down the 6 observability patterns from Microservices Patterns by Chris Richardson — the same patterns used in production at companies like Netflix, Uber, and Discord. By the end, you'll know exactly which pattern to reach for and when.

Why Observability Is a Microservices Problem

In a monolith, when something goes wrong you open one log file. One database. One process. The answer is usually in there.

In a microservices architecture, a single user request can touch a dozen services. Each service has its own database, its own logs, its own metrics. The classic FTGO food delivery app from Richardson's book illustrates this perfectly — a single getOrderDetails() call hits the API gateway, Order Service, Kitchen Service, Delivery Service, and Accounting Service. That's 5 log files to correlate manually, unless you have the right observability infrastructure in place.

"A service instance can fail without terminating. A bug might cause an instance to run out of database connections and be unable to access the database — yet still be running."

There are 6 patterns that solve this. They divide cleanly into a developer responsibility and an operations responsibility. Developers implement the signal. Operations builds the infrastructure to collect it.

The 6 Observability Patterns

Pattern 01
Health Check API

Expose an endpoint that returns whether the service is alive and ready to handle requests.

Pattern 02
Log Aggregation

Ship logs from all services into a centralized server with search, filtering, and alerting.

Pattern 03

Tag each request with a unique ID and track its journey across every service it touches.

Pattern 04

Pattern 1: Health Check API

A service can be running but completely broken. It might have exhausted its database connection pool, or its messaging adapter might have failed. Without a health check, the deployment infrastructure keeps routing requests to a broken instance — and users feel the pain.

The fix is simple: expose a GET /health endpoint that tests the service's actual connections and returns a status code. 200 = healthy. 503 = unhealthy.

Real world

Spring Boot Actuator implements GET /actuator/health out of the box. It auto-configures checks based on what's on your classpath — JDBC DataSource, RabbitMQ, Redis. Kubernetes uses this endpoint for both readinessProbe (should I route traffic here?) and livenessProbe (should I restart this pod?).

The key insight: a health check should test real dependencies, not just return 200 unconditionally. A database that's down should make your health check fail.

Pattern 2: Log Aggregation

Logs scattered across dozens of service instances are useless for debugging distributed requests. You need them all in one place.

The popular stack is ELK: Elasticsearch (storage + search), Logstash (pipeline), Kibana (visualization). Alternatives include Fluentd, AWS CloudWatch Logs, and Datadog.

The developer's job is simpler than you think: log to stdout. Don't write to files. The deployment infrastructure (Docker, Kubernetes) picks up stdout and routes it to the log pipeline. This is especially important in containerized environments where there's no persistent filesystem to write to.

Pro tip

Always log in structured JSON. Plain text logs are hard to query. A JSON log entry like {"level":"ERROR","service":"order-service","traceId":"8d8f...","msg":"DB timeout"} lets you filter by any field instantly in Kibana.

Pattern 3: Distributed Tracing

This is the most powerful pattern for debugging latency and cross-service failures. When a request enters your system, assign it a unique trace ID. Every service that handles that request stamps this ID onto its logs and passes it to the next service via HTTP headers.

The result: you can see the complete journey of any request — which services it touched, how long each took, where it failed.

A trace for GET /orders/1 → API Gateway → Order Service
Span ABC API Gateway — 45ms total
Span DEF Order Service — 32ms
Span GHI MySQL query — 18ms

The trace propagation uses standardized headers. The B3 standard (used by Zipkin) propagates X-B3-TraceId and X-B3-ParentSpanId between services. AWS X-Ray uses its own header format.

A huge bonus: when you combine distributed tracing with log aggregation, you can search your logs by trace ID and instantly find every log line generated by a single user request — across all services.

// Log entry with trace context injected by Spring Cloud Sleuth
INFO [ftgo-order-service,8d8fdc37be104cc6,8d8fdc37be104cc6,false]
                          ↑ traceId           ↑ spanId

// Search all logs for this trace ID → find every service involved
Framework pick

Spring Cloud Sleuth auto-instruments distributed tracing using AOP — you add it as a dependency and it handles header propagation automatically. For the tracing server, Zipkin (open source) or AWS X-Ray (managed) are the standard choices.

Pattern 4: Exception Tracking

Exceptions in log files are problematic for three reasons: log files are line-oriented (stack traces span many lines), there's no built-in deduplication (100 instances of the same bug = 100 separate log entries), and there's no tracking of resolution status.

An exception tracking service solves all three. Services report exceptions via a REST API. The service de-duplicates, generates alerts, and provides a UI to manage resolution. Examples: Sentry (open source + cloud), HoneyBadger, Rollbar.

// Using Sentry in Java
try {
  orderService.createOrder(orderDetails);
} catch (Exception e) {
  Sentry.captureException(e);  // reports to Sentry with full context
  throw e;
}

// Or use a Servlet Filter to catch everything automatically

Pattern 5: Application Metrics

Health checks tell you if a service is alive. Metrics tell you how well it's doing. Business-level metrics — orders placed per minute, payment success rate, cart abandonment — give you early warning of problems before users start complaining.

The standard Java approach is Micrometer, which provides a vendor-neutral API. Configure it to export to Prometheus (pull model) or CloudWatch (push model).

// Instrument OrderService with Micrometer
public class OrderService {
  private MeterRegistry meterRegistry;

  public Order createOrder(...) {
    // ... business logic ...
    meterRegistry.counter("placed_orders").increment();
    return order;
  }

  public void approveOrder(long orderId) {
    // ... business logic ...
    meterRegistry.counter("approved_orders").increment();
  }
}

Pattern 6: Audit Logging

Audit logging records who did what to which entity and when. It's used for compliance (GDPR, SOC2), customer support ("what happened to order 1234?"), and security (detecting suspicious behavior).

There are 3 implementation approaches with different tradeoffs:

Option A: Inline audit logging

Add audit log writes directly inside business logic methods. Simple but couples auditing to business code and risks being forgotten by developers.

Option B: AOP-based audit logging

Use Spring AOP to intercept service methods and automatically write audit entries. More reliable — no developer can forget it. Tradeoff: the advice only sees method names and arguments, so generating human-readable audit entries can require extra work.

Option C: Event sourcing

If you're using event sourcing, audit logging comes for free. Every state change is stored as an immutable event. Your event store is your audit log — with full history, rollback capability, and temporal queries included.

Which to choose?

For new services: use AOP if you're not on event sourcing. For event-sourced services: leverage the built-in audit trail. Inline code is a last resort for one-off cases with complex domain context requirements.

Putting It All Together: The Microservice Chassis

Setting up all 6 patterns from scratch for every new service would take days. The solution is a microservice chassis — a framework that handles these cross-cutting concerns so you can focus on business logic from day one.

For Java, the chassis is Spring Boot + Spring Cloud. Add the right dependencies, and you get: health checks via Actuator, distributed tracing via Sleuth, metrics via Micrometer, and structured logging via SLF4J + Logback — all configured automatically.

The emerging alternative is a service mesh (Istio, Linkerd), which moves networking-level observability — distributed tracing propagation, load balancing, circuit breaking — out of the service and into the infrastructure layer. This makes the language/framework irrelevant for many concerns.

TL;DR — What to remember
  • Health Check API — expose /health and test real dependencies
  • Log Aggregation — log to stdout in JSON, ship to ELK or CloudWatch
  • Distributed Tracing — propagate trace IDs, use Zipkin or X-Ray
  • Exception Tracking — use Sentry or HoneyBadger, not raw log alerts
  • Application Metrics — Micrometer → Prometheus → Grafana
  • Audit Logging — AOP or event sourcing, never leave it optional
  • Every pattern has a developer half and an ops half — own your half
